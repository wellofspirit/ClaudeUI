# Writing Patch READMEs

## Purpose

Each patch in `patch/` modifies `cli.js` — a minified ~11MB single-file bundle where **every function name changes on every SDK release**. The README.md is NOT user documentation. It is a **reverse-engineering survival guide** that enables a future agent (with zero prior context) to:

1. Understand why the patch exists and whether it's still needed
2. Verify the bug persists in a new SDK version
3. Locate the same code sites when all minified names are different
4. Rebuild `apply.mjs` from scratch
5. Avoid re-investigating dead ends and syntax pitfalls already discovered

**Write for stale names.** Every minified function name in the document WILL be wrong in the next SDK version. The value is in architecture descriptions, code patterns, search techniques, and especially the discovery narrative — not the specific names.

## Before Writing

1. **Read the patch's `apply.mjs`** — understand every regex, every injection, every safety check
2. **Read the existing READMEs** in `patch/` for style and depth reference:
   - `subagent-streaming/README.md` — exemplar for multi-patch coordination, architecture diagrams, message flow
   - `team-streaming/README.md` — exemplar for syntax pitfall documentation, comma-vs-semicolon analysis
   - `mcp-tool-refresh/README.md` — exemplar for two-path comparison (CLI React vs SDK), data flow diagrams
   - `queue-control/README.md` — exemplar for concise single-concern patches
3. **Use `/bundle-analyzer`** to verify your understanding of the code sites

## Required Sections

Every patch README MUST include all of these sections. The order below is the canonical order.

### 1. Title + One-Line Summary

```markdown
# Patch: <name>

<One sentence describing the user-visible bug this fixes.>
```

### 2. Affected Component

- State the exact file: `@anthropic-ai/claude-agent-sdk` — bundled `cli.js`
- Version table: SDK package version + bundled CLI version at time of discovery
- Note that the SDK bundles its own CLI, independent of the native `claude` binary

```markdown
## Affected Component

`@anthropic-ai/claude-agent-sdk` — bundled `cli.js` file.

| Component | Version at time of discovery |
|---|---|
| SDK package | 0.2.XX |
| Bundled CLI (`cli.js`) | 2.1.XX |
```

### 3. The Problem(s)

- **User-visible symptom** — what's broken, missing, or invisible. Be concrete.
- **Root cause** — why the code behaves this way. Trace from symptom to code.
- If multiple problems, number them clearly.

Bad: "Messages don't work properly."
Good: "Teammate thinking tokens, text, tool calls, and stream deltas never reach the SDK consumer's stdout. The UI shows nothing while the teammate is working."

### 4. Architecture Overview

This is the most important section for rebuilding. Explain the system, not just the patch site.

- **ASCII diagrams** showing data flow through the relevant components
- **Key function signatures** with destructured parameters (real minified names from current version)
- **Variable mapping table** — what each variable is, where it comes from, what it holds
- **How the two paths differ** (if applicable) — e.g., CLI React UI vs SDK headless, sync vs async, foreground vs background
- **Why one path works and the other doesn't** — the architectural reason for the bug

Example of a good variable table:

```markdown
| Variable | Source | Value |
|---|---|---|
| `q` | `A.identity` | `{ agentId: "name@team", agentName: "name", ... }` |
| `K` | `A.taskId` | `"tXXXXXX"` (generated by `vB("in_process_teammate")`) |
| `M` | `H.setAppState` | AppState setter function |
```

Example of a good two-path comparison:

```markdown
### CLI React UI path (works)
refreshTools: () => { let state = r.getState(); return Zf6(...) }
→ called after each turn in iR → tools updated live

### SDK query path (broken)
options.tools = z  (frozen at query creation time)
options.refreshTools = undefined  ← BUG
→ refreshTools check skipped → tools never update
```

### 5. The Patches (one subsection per patch letter)

For each patch (A, B, C, etc.):

#### Marker
```markdown
**Marker**: `/*PATCHED:<name>-<letter>*/`
```

#### Anchor (the unique string that locates the injection site)
```markdown
### Anchor (unique, 1 match)
\`\`\`
<exact string from cli.js that matches exactly once>
\`\`\`
```

#### Before / After code
Show the exact transformation. Use real minified variable names. Add comments for clarity.

```markdown
### Before
\`\`\`js
override:{abortController:T}
\`\`\`

### After
\`\`\`js
/*PATCHED:team-streaming-A*/override:{abortController:T,agentId:q.agentId.replace(/@/g,"--")}
\`\`\`
```

#### Why it's safe
Explain why this doesn't break anything:
- Other callers of the same function
- Type validation / Zod schemas
- Interactive vs headless mode
- Performance implications
- Existing downstream handlers

#### Dynamic function extraction (if applicable)
If the apply.mjs extracts minified names via regex, document the extraction pattern:

```markdown
### Dynamic function extraction

`SES` (session ID) and `UUID` (unique ID generator) are extracted at apply time:
\`\`\`js
const sessFnRe = /session_id:([\w$]+)\(\).*?parent_tool_use_id/
const uuidFnRe = /\{type:"progress",data:[\w$]+,...,uuid:([\w$]+)\(\),timestamp:new Date/
\`\`\`
```

### 6. Message Format (if the patch emits new messages)

Show the exact JSON shape written to stdout. Include all fields.

```markdown
### Message format
\`\`\`json
{"type":"stream_event","event":{...},"teammate_id":"name@team","session_id":"...","uuid":"..."}
\`\`\`
```

If there's a Zod schema that validates the message, document it:

```markdown

\`\`\`js

  type: b.literal("system"),
  subtype: b.literal("task_notification"),
  task_id: b.string(),
  status: b.enum(["completed", "failed", "stopped"]),

\`\`\`
```

### 7. How to Find This Code

Provide `bundle-analyzer` commands for every code site. This is what future you will use when all names have changed.

```markdown
## How to Find This Code

### <Function name> (<purpose>)
\`\`\`bash
bundle-analyzer find cli.js "<unique string literal>" --compact
\`\`\`
```

Group related searches together. Include multiple search strategies (primary + fallback) when possible.

### 8. Syntax Pitfalls

Document every syntax trap encountered during development. These are the most expensive mistakes to re-discover.

**Critical pitfalls to watch for:**

- **Missing semicolons after `im()` / function calls ending with `)`**: ASI does NOT insert `;` before `if` after `)` — it's a parse error, not an ASI case
- **Semicolons in comma expressions**: `return A, B, C` — using `;` instead of `,` terminates the return and creates dead code with label-syntax errors
- **`continue` vs collection**: `stream_event` messages lack `.message`/`.uuid` — pushing them to collection arrays or passing to stats functions crashes

Format each pitfall with WRONG and CORRECT examples:

```markdown
### Pitfall: Missing semicolons after function calls

\`\`\`js
// WRONG — SyntaxError: Unexpected token 'if'
im(K, fn, M)if(_6.type === "assistant") ...

// CORRECT
im(K, fn, M);if(_6.type === "assistant") ...
\`\`\`

ASI does NOT help here because...
```

Always end with: **"Always run `node --check cli.js` after applying patches."**

### 9. What's NOT Changed

Explicitly list things you investigated but intentionally left alone:

```markdown
## What's NOT Changed

**Collection arrays** — `x.push(_6)` and `N.push(_6)` still operate on
assistant/user messages. Only `stream_event` skips them (via `continue`).

**im() state updates** — Still run before forwarding. AppState is always up-to-date.
```

This prevents future agents from re-investigating the same decisions.

### 10. Consumer-Side Integration (if applicable)

If the patch requires changes in `claude-session.ts` or the renderer to be useful, document the consumer-side wiring:

- How the consumer detects/routes the new messages
- What maps/stores are involved
- The full round-trip from patch stdout → SDK → IPC → renderer

### 11. Verification

Concrete steps to verify the patch works:

```markdown
## Verification

1. `node patch/<name>/apply.mjs` — should apply all patches
2. Run again — should report "already applied"
3. `node --check node_modules/@anthropic-ai/claude-agent-sdk/cli.js` — no syntax errors
4. `node patch/apply-all.mjs` — all patches pass
5. <Manual test steps specific to this patch>
```

### 12. Discovery Method

**This is the second most important section after Architecture Overview.** It teaches future agents HOW to investigate, not just WHAT was found.

Write as a numbered narrative — the detective story of how you found each piece:

```markdown
## Discovery Method

1. **Observed the symptom**: <what was visible/broken>
2. **Traced to code site**: <how you found the relevant function>
3. **Identified root cause**: <what the code does wrong and why>
4. **First fix attempt (failed)**: <what didn't work and why>
5. **Correct fix**: <what works and the insight that led to it>
6. **Verified full round-trip**: <how you confirmed end-to-end>
```

Include failed attempts! They prevent future agents from repeating the same mistakes. The B2 semicolon bug in team-streaming and the Part-A-only insufficient fix in mcp-tool-refresh are perfect examples.

## Writing Principles

### Depth over Brevity

A patch README should be 200-700 lines. The `team-streaming` README is 724 lines and covers 3 sub-patches with full architecture analysis. The `mcp-status` README is 84 lines for a simple 2-part fix. Scale depth to complexity, but never skip sections — even short patches benefit from architecture context.

### Exact Code Over Paraphrasing

Always copy the exact minified code with comments. Future agents need to pattern-match against real code:

```js
// GOOD — exact code with inline explanation
return im(K, (T) => ({...T, status:"completed", endTime:Date.now()}), M),
       {success: !0, messages: N}
// This is a comma expression: evaluate im(...), discard result, return {success:!0,...}

// BAD — paraphrased
// The function updates state and returns a success object
```

### Search Patterns Over Names

For every code site, provide a search pattern that works across versions:

```
// GOOD — content-based pattern (survives name changes)
bundle-analyzer find cli.js "override:{abortController:T}" --compact

// BAD — name-based (breaks on next version)
bundle-analyzer find cli.js "function KfY" --compact
```

Exception: it's fine to include name-based searches as secondary/convenience options alongside content-based ones.

### Data Flow Diagrams

Use ASCII diagrams showing the complete path from data origin to consumer. Mark where each patch intervenes:

```
User types prompt → InputBox → IPC → session.run()
  → sdkQuery() → cli.js subprocess
    → stream_event → [Patch B intercepts here] → stdout → SDK → IPC → renderer
    → assistant → [Patch B2 forwards here] → stdout → SDK → IPC → renderer
```

### Before/After Flow Diagrams (for complex patches)

Show the complete flow both before and after patching:

```markdown
### Before patch (broken)
User sends message → EVq({ tools: w6 }) → w6 frozen at session start ❌

### After patch (fixed)
User sends message → Part A refreshes w6 → EVq({ tools: w6 }) → fresh tools ✅
```

### Version Tracking

When a patch spans multiple SDK versions, track what changed:

```markdown
| What | v2.1.45 | v2.1.49 | v2.1.59 |
|---|---|---|---|
| Bash progress check | `bash_progress` only | `bash_progress \|\| powershell_progress` | same |
| Status validator | manual | includes `killed` natively | same |
```

## Anti-Patterns to Avoid

1. **Don't skip the Architecture section** — "the bug is at line X, we change Y" is useless when line X doesn't exist in the next version
2. **Don't omit failed attempts** — they're the most valuable part of Discovery Method
3. **Don't use only minified names for search** — always provide content-pattern-based search strategies
4. **Don't forget Zod schemas** — if the patch emits new message types, verify against the SDK's Zod validation
5. **Don't ignore the consumer side** — a patch that writes to stdout is useless if nothing reads it
6. **Don't skip "What's NOT Changed"** — this prevents wasted investigation cycles
7. **Don't assume the reader knows the SDK** — explain what `im()` does, what `stream_event` is, why comma expressions matter

## Checklist Before Finalizing

- [ ] Every code site has a `bundle-analyzer find` command
- [ ] Every anchor string verified to have exactly 1 match in cli.js
- [ ] Before/after code uses real minified names (not pseudocode)
- [ ] Variable mapping table covers all variables used in injected code
- [ ] Syntax pitfalls documented with WRONG/CORRECT examples
- [ ] `node --check` mentioned as validation step
- [ ] Discovery Method includes failed attempts
- [ ] Related patches cross-referenced
- [ ] Message format documented (if patch emits new messages)
- [ ] Consumer-side integration documented (if applicable)
