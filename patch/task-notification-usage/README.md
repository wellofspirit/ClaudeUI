# Patch: task-notification-usage (RETIRED)

> **Upstreamed in SDK 0.2.49 (CLI 2.1.49).** This patch is no longer applied.
> The SDK now natively extracts `<usage>` data (total_tokens, tool_uses,
> duration_ms) from task-notification XML and includes `killed → stopped`
> status mapping. Kept for historical reference.

Adds `<usage>` extraction to the `{type:"system", subtype:"task_notification"}`
message emitted by the SDK streaming loop. Without this patch, background agent
completion notifications lack cost/token/duration data.

## Affected Component

`@anthropic-ai/claude-agent-sdk` — bundled `cli.js` file (same file patched
by `task-notification` and `subagent-streaming`).

| Component | Version at time of discovery |
|---|---|
| SDK package | 0.2.39 |
| Bundled CLI (`cli.js`) | 2.1.39 |
| Upstreamed in | SDK 0.2.49 / CLI 2.1.49 |

## The Bug

When a background agent completes, the CLI generates a `<task-notification>`
XML block that includes `<usage>` data (total_tokens, tool_uses, duration_ms).
This XML is processed by the streaming loop's task-notification handler, which
parses `<task-id>`, `<output-file>`, `<status>`, and `<summary>` — but
**ignores `<usage>`**.

The system message emitted to the SDK consumer contains no usage data. The
full XML (including `<usage>`) is then fed to the model as a prompt for the
next API call, but it's never yielded as a user message to the SDK consumer.

### The XML contains usage

Generated by `BK1()` (background agent completion handler):

```xml
<task-notification>
<task-id>abc123</task-id>
<status>completed</status>
<summary>Agent "name" completed</summary>
<result>...</result>
<usage>total_tokens: 5432
tool_uses: 12
duration_ms: 8500</usage>
</task-notification>
```

### The streaming loop drops usage

```js
// Line ~355654 in beautified cli.js (char ~10825200 in minified)
if (D1.mode === "task-notification") {
    let Y1 = typeof D1.value === "string" ? D1.value : "",
        w1 = Y1.match(/<task-id>([^<]+)<\/task-id>/),
        P1 = Y1.match(/<output-file>([^<]+)<\/output-file>/),
        X1 = Y1.match(/<status>([^<]+)<\/status>/),
        f1 = Y1.match(/<summary>([^<]+)<\/summary>/),
        // ← NO <usage> extraction here
        ...;
    P.enqueue({
        type: "system",
        subtype: "task_notification",
        task_id: w1?.[1] ?? "",
        status: G1,
        output_file: P1?.[1] ?? "",
        summary: f1?.[1] ?? "",
        // ← NO usage field
        session_id: p6(),
        uuid: SE()
    })
}
// Then: let o = D1.value  ← full XML used as prompt (never yielded)
```

### Why the user message path doesn't help

After emitting the system message, the streaming loop uses the full XML as
the `prompt` parameter for the next inner query (`rMq()`). This prompt is
converted into an internal user message and added to `mutableMessages`, but
it is **never yielded** to the outer SDK consumer. The SDK consumer only
receives the system message (without usage) and the model's subsequent
assistant response.

## The Fix

Extract `<usage>` from the XML and include it in the system message.

### Before

```js
P.enqueue({
    type: "system", subtype: "task_notification",
    task_id: w1?.[1] ?? "", status: G1,
    output_file: P1?.[1] ?? "", summary: f1?.[1] ?? "",
    session_id: p6(), uuid: SE()
})
```

### After

```js
var _ub = Y1.match(/<usage>([\s\S]*?)<\/usage>/),
    _udata = _ub ? {
        total_tokens: +((_ub[1].match(/total_tokens:\s*(\d+)/) || [])[1] || 0),
        tool_uses: +((_ub[1].match(/tool_uses:\s*(\d+)/) || [])[1] || 0),
        duration_ms: +((_ub[1].match(/duration_ms:\s*(\d+)/) || [])[1] || 0)
    } : null;
P.enqueue({
    type: "system", subtype: "task_notification",
    task_id: w1?.[1] ?? "", status: G1,
    output_file: P1?.[1] ?? "", summary: f1?.[1] ?? "",
    usage: _udata,      // ← NEW
    session_id: p6(), uuid: SE()
})
```

The `usage` field is `null` when no `<usage>` block is present (e.g., failed
or stopped agents that didn't accumulate usage).

## Companion Change

`src/main/services/claude-session.ts` reads `msg.usage` from the system
notification and converts it to `{ totalTokens, toolUses, durationMs }` for
the `TaskNotification` type. This replaces the previous approach of trying
to extract usage from the user message XML (which never arrives).

## Applying the Patch

```bash
node patch/task-notification-usage/apply.mjs
```

The script:

1. Finds `cli.js` in the SDK package
2. Locates the `<summary>` regex match (unique anchor in the codebase)
3. Finds the `P.enqueue({type:"system",subtype:"task_notification",...})`
   call that follows
4. Injects `<usage>` extraction before the enqueue
5. Adds `usage: _udata` to the enqueued object
6. Verifies the patch was applied

### Re-applying after SDK updates

```bash
node patch/apply-all.mjs
```

The script is idempotent — detects if already applied and skips.

## How to find this code in a new version

1. Search for `match(/<summary>([^<]+)<\/summary>/)` — unique in the codebase
2. The `P.enqueue` with `subtype:"task_notification"` follows within ~200 chars
3. Variable names will change but the structure is stable
