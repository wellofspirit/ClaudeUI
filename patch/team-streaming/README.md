# Patch: team-streaming

Fixes three bugs with in-process teammates (spawned via TeamCreate + Task
with `team_name`):

1. **JSONL fragmentation** — Each turn generates a random agentId, creating
   a separate `agent-<hex>.jsonl` per turn instead of one file per teammate.
2. **No event streaming** — Teammate thinking, text, tool calls, and stream
   deltas never reach the SDK consumer's stdout.
3. **No completion notification** — When a teammate finishes or fails, no
   `task_notification` is emitted. The SDK consumer never learns the
   teammate stopped, so the UI shows "running" forever.

## Affected Component

`@anthropic-ai/claude-agent-sdk` — bundled `cli.js` file.

| Component | Version at time of discovery |
|---|---|
| SDK package | 0.2.45 |
| Bundled CLI (`cli.js`) | 2.1.45 |

## The Problems

### Problem 1: JSONL Fragmentation

When the SDK runs an in-process teammate via `KfY`, each AI turn calls
`OR()` (the query runner) to get an async iterator of messages. Inside
`OR()`:

```js
let N = O?.agentId ? O.agentId : Qy()
```

`Qy()` generates a random hex string. `KfY` passes
`override:{abortController:T}` — no `agentId`. So every turn gets a new
random ID, producing `agent-a20745a.jsonl`, `agent-b38cf91.jsonl`, etc.

A 2-agent 3-round debate produces 14 JSONL files instead of 2. This makes
session history reload nearly impossible and forced a 600-line workaround
(`subagent-watcher.ts`) with fuzzy name-keyword matching.

### Problem 2: No Event Streaming

Inside `KfY`'s `for await` loop, messages are collected into arrays and
state is updated via `im()`, but nothing writes to stdout. The progress
callback (`j`) is not called here — it's only wired for the sync Task
tool path. So teammate stream_events, assistant messages, and user
(tool_result) messages are invisible to the SDK consumer.

### Problem 3: No Completion Notification

When `KfY` finishes (success or failure), it updates in-memory AppState
via `im()`:

```js
// Success path
return im(K, (T) => ({...T, status: "completed", endTime: Date.now()}), M),
       {success: !0, messages: N}

// Failure path (in catch block)
im(K, (B) => ({...B, status: "failed", error: R, ...}), M),
sm4(q.agentName, q.color, q.teamName, {...}),
{success: !1, error: R, messages: N}
```

**Neither path writes a `task_notification` to stdout.** The headless
mode's notification code (at char ~10809315 in v2.1.45) only processes
XML-based `<task-notification>` messages from background tasks. In-process
teammates bypass that entirely — their lifecycle is managed by `im()`
(in-memory state updates) which the SDK consumer cannot observe.

This means the SDK consumer's `taskIdMap` lookup at
`this.taskIdMap.get(taskId)` never fires for teammates, so the UI never
updates their status from "running" to "completed"/"failed".

## Architecture Overview

### KfY — In-process teammate runner

```js
async function KfY(A) {
    let { identity: q, taskId: K, prompt: Y, description: z,
          agentDefinition: w, teammateContext: $, toolUseContext: H,
          abortController: O, model: _, ... } = A
    let { setAppState: M } = H
    //  q.agentId = "ts-advocate@lang-debate" (stable name@team identity)
    //  K = taskId (generated by vB("in_process_teammate") → "t" + 6 random hex chars)

    // ... setup: system prompt, tools, history array N[] ...

    try {
        while (!O.signal.aborted && !v) {
            // Each iteration is one AI turn for this teammate
            for await (let _6 of OR({
                agentDefinition: A6,
                promptMessages: B,
                toolUseContext: H,
                // ...
                override: { abortController: T },  // ← Patch A injects agentId here
                // ...
            })) {
                // Patch B1 injects stream_event bypass here (before push)
                x.push(_6), N.push(_6), R06(p, _6, Q, H.options.tools)

                im(K, (k6) => ({
                    // ... update appState with token counts, tool results ...
                    lastReportedTokenCount: VP8(p)
                }), M)
                // Patch B2 injects assistant/user forwarding here (after im())
            }

            // After for-await: wait for next message from inbox
            let w6 = await qfY(q, O, K, H.getAppState, M, q.parentSessionId)
            // ... handle shutdown_request, new_message, aborted ...
        }

        // Patch C1 injects task_notification here (success)
        return im(K, (T) => ({...T, status:"completed", endTime:Date.now()}), M),
               {success: !0, messages: N}

    } catch (T) {
        // Patch C2 injects task_notification here (failure)
        return im(K, (B) => ({...B, status:"failed", error:R, ...}), M),
               sm4(q.agentName, q.color, q.teamName, {...}),
               {success: !1, error: R, messages: N}
    }
}
```

### OR — Query runner (creates JSONL file)

```js
async function* OR({ ..., override: O, ... }) {
    let N = O?.agentId ? O.agentId : Qy()  // ← falls through to random hex
    // N flows to: lh(cZ(N)) → "agent-${N}.jsonl" → fs.appendFile
}
```

Creates the task entry in AppState and returns `{ taskId, agentId, ... }`:

    return { success: true, agentId: _, taskId: J, ... }

### How the Task tool returns teammate info

When the Task tool spawns an in-process teammate, the tool result text
(visible to the model and parsed by `detectTaskMapping`) looks like:

```
Spawned successfully.
agent_id: ts-advocate@lang-debate
name: ts-advocate
team_name: lang-debate
The agent is now running and will receive instructions via mailbox.
```

`detectTaskMapping` uses `AGENT_ID_RE = /(?:agentId|agent_id):\s*(\S+)/`
to extract `ts-advocate@lang-debate` and stores it in `taskIdMap`:

```
taskIdMap["ts-advocate@lang-debate"] → "toolu_013g6v5JN86ZLyaFyEwESiBE"
```

### How task_notification is consumed

In `claude-session.ts`, when a `task_notification` system message arrives:

```js
const taskId = msg.task_id        // e.g., "ts-advocate@lang-debate"
const matched = this.taskIdMap.get(taskId)  // → toolUseId
if (matched) {
    this._teammateStatuses.set(matched, statusMap[taskStatus])
    this.send('session:task-notification', { toolUseId: matched, status, ... })
}
```

The renderer receives this via IPC and calls `updateTeammateStatus()` in
the Zustand store, which updates the UI indicator.

### Message types in the for-await loop

| Type | Has `.message`? | Has `.uuid`? | Safe for R06/im()? |
|---|---|---|---|
| `stream_event` | No | No | **No** — breaks if pushed to arrays |
| `assistant` | Yes | Yes | Yes |
| `user` | Yes | Yes | Yes |
| `progress` | No | Yes | Handled separately |

### Why background tasks work but teammates don't

Background tasks (`local_agent` type) run via a different code path:

1. The headless mode (`m3z`/`f6`) processes `<task-notification>` XML from
   the SDK's message stream
2. This XML is generated by the background agent's exit handler
3. The XML is parsed and enqueued as a `task_notification` system message

In-process teammates (`in_process_teammate` type) run inside `KfY` in
the same process. They update AppState directly via `im()`. No XML is
generated, no system message is enqueued, no stdout is written.

## Patch A: Fix agentId Fragmentation

**Marker**: `/*PATCHED:team-streaming-A*/`

### Anchor (unique, 1 match)

```
override:{abortController:T}
```

### Before

```js
override:{abortController:T}
```

### After

```js
/*PATCHED:team-streaming-A*/override:{abortController:T,agentId:q.agentId.replace(/@/g,"--")}
```

The `replace(/@/g,"--")` sanitizes `@` for filenames (Windows-unsafe).
The agentId flows through `lh(cZ(id))` → `agent-${id}.jsonl`, producing
`agent-ts-advocate--lang-debate.jsonl` instead of `agent-a20745a.jsonl`.

### Why this is safe

- `q.agentId` is the stable `"name@team"` identity, already in scope as
  the `identity` parameter of `KfY`.
- `appendEntry` uses `fs.appendFile` — appends, never truncates. Multiple
  turns append to the same file.
- UUID dedup is per-session, no collision risk across turns.
- The `@` → `--` transform is reversible and produces valid filenames on
  all platforms.

## Patch B: Forward Teammate Events to Stdout

**Marker**: `/*PATCHED:team-streaming-B*/`

Two injection points inside the `for await` loop body.

### Dynamic function extraction

`SES` (session ID) and `UUID` (unique ID generator) are extracted from
the bundle at apply time:

```js
const sessFnRe = /session_id:([\w$]+)\(\).*?parent_tool_use_id/
const uuidFnRe = /\{type:"progress",data:[\w$]+,toolUseID:[\w$]+,parentToolUseID:[\w$]+,uuid:([\w$]+)\(\),timestamp:new Date/
```

Currently `p1()` and `Gf()` respectively. These same regexes are used
in Patch E of subagent-streaming.

### B1: Stream event bypass (before collection arrays)

**Anchor** (unique, 1 match):

```
x.push(_6),N.push(_6),R06(p,_6,Q,H.options.tools)
```

**Injection** (prepended before anchor):

```js
/*PATCHED:team-streaming-B*/if(_6.type==="stream_event"){
  process.stdout.write(JSON.stringify({
    type:"stream_event",event:_6.event,
    teammate_id:q.agentId,
    session_id:SES(),uuid:UUID()
  })+"\n");continue}
```

`stream_event` messages lack `.message`/`.uuid` and would break `R06()`
/ `im()`. The `continue` skips collection entirely (same pattern as
Patch F in subagent-streaming).

### B2: Assistant/user forwarding (after im() update)

**Anchor** (unique, 1 match):

```
lastReportedTokenCount:VP8(p)}},M)
```

**Injection** (appended after anchor, with leading `;`):

```js
;if(_6.type==="assistant"||_6.type==="user")
  process.stdout.write(JSON.stringify({
    type:_6.type,message:_6.message,
    teammate_id:q.agentId,
    session_id:SES(),uuid:UUID()
  })+"\n");
```

**CRITICAL**: The leading `;` is required. The anchor ends with `)` from
the `im(K, ..., M)` call. Without a `;`, the parser tries to interpret
`im(...)if(...)` as a single expression, which is a `SyntaxError`. ASI
does NOT insert a semicolon before `if` in this context because the
preceding `)` could theoretically be followed by more of the expression.
This was the cause of a "Query closed before response received" crash
during initial development — the subprocess failed `node --check` and
exited immediately.

This runs AFTER the existing `im()` update, so AppState is already
updated before we forward.

### Message format

```json
{"type":"stream_event","event":{...},"teammate_id":"ts-advocate@lang-debate","session_id":"...","uuid":"..."}
{"type":"assistant","message":{...},"teammate_id":"ts-advocate@lang-debate","session_id":"...","uuid":"..."}
{"type":"user","message":{...},"teammate_id":"ts-advocate@lang-debate","session_id":"...","uuid":"..."}
```

Uses `teammate_id` (the unsanitized `name@team` identity) instead of
`parent_tool_use_id`. The consumer maps this to the detected teammate's
`toolUseId` via `teammateIdToToolUse` map in `claude-session.ts`.

## Patch C: Teammate Completion/Failure Notifications

**Marker**: `/*PATCHED:team-streaming-C*/`

Two injection points at KfY's exit paths. Both are **comma expressions**
(`return A, B, C`) where the last element is the return value. We insert
the `process.stdout.write()` call as an additional comma-separated element.

### Why comma expressions, not semicolons

KfY's success path is:

```js
return im(K, fn, M), {success: !0, messages: N}
```

This is a comma expression: evaluate `im(...)`, discard its result,
return `{success: !0, messages: N}`. If we insert a `;` instead of `,`,
we'd break this into two statements:

```js
return im(K, fn, M), process.stdout.write(...);
{success: !0, messages: N}  // ← unreachable, and `:` is a SyntaxError
```

The `;` terminates the `return`, making `{success:!0,messages:N}` a
dead code block where `success:!0` is parsed as a label, and
`messages:N` causes `Unexpected token ':'`. This was caught by
`node --check` during development.

### C1: Success path

**Anchor** (unique, 1 match):

```
status:"completed",endTime:Date.now()}),M),{success:!0,messages:N}
```

**Replacement** (anchor is replaced entirely):

```js
/*PATCHED:team-streaming-C*/status:"completed",endTime:Date.now()}),M),
process.stdout.write(JSON.stringify({
  type:"system",subtype:"task_notification",
  task_id:q.agentId,status:"completed",
  output_file:"",summary:"",
  session_id:SES(),uuid:UUID()
})+"\n"),
{success:!0,messages:N}
```

The full expression becomes: `return im(...), notify(), {success:!0,...}`

### C2: Failure path

**Anchor** (unique, 1 match):

```
endTime:Date.now(),onIdleCallbacks:[]}},M),sm4(
```

**Replacement** (anchor is replaced entirely):

```js
endTime:Date.now(),onIdleCallbacks:[]}},M),
process.stdout.write(JSON.stringify({
  type:"system",subtype:"task_notification",
  task_id:q.agentId,status:"failed",
  output_file:"",summary:"",
  session_id:SES(),uuid:UUID()
})+"\n"),
sm4(
```

The full expression becomes: `return ..., im(...), notify(), sm4(...), {success:!1,...}`

### task_notification format

```json
{
  "type": "system",
  "subtype": "task_notification",
  "task_id": "ts-advocate@lang-debate",
  "status": "completed",
  "output_file": "",
  "summary": "",
  "session_id": "...",
  "uuid": "..."
}
```

The `task_id` is `q.agentId` (e.g., `"ts-advocate@lang-debate"`), which
matches the key stored in `taskIdMap` by `detectTaskMapping()` when
the Task tool result arrived.

The `SGY` Zod schema (char ~8072331) validates this format:
```js
b.object({
  type: b.literal("system"),
  subtype: b.literal("task_notification"),
  task_id: b.string(),
  status: b.enum(["completed", "failed", "stopped"]),
  output_file: b.string(),
  summary: b.string(),
  uuid: pJ(),
  session_id: b.string()
})
```

## Syntax Pitfalls

### Pitfall 1: Missing semicolons after `im()` calls

The `im(K, fn, M)` call ends with `)`. If the next statement starts
with `if(`, you MUST insert a `;` between them:

```js
// WRONG — SyntaxError: Unexpected token 'if'
im(K, fn, M)if(_6.type === "assistant") ...

// CORRECT
im(K, fn, M);if(_6.type === "assistant") ...
```

ASI does NOT help here. The parser doesn't insert a semicolon before
`if` after `)` because `if` is not a valid expression continuation —
it's simply a parse error.

### Pitfall 2: Semicolons in comma expressions

When injecting into `return A, B, C` comma expressions, use `,` not `;`:

```js
// WRONG — SyntaxError: Unexpected token ':'
return im(...), process.stdout.write(...); {success:!0, messages:N}
//                                       ^ terminates return
//                                         {success:!0, messages:N} is dead code
//                                         messages:N → unexpected ':'

// CORRECT
return im(...), process.stdout.write(...), {success:!0, messages:N}
```

## Consumer-Side Integration

### TeamDelete detection

The SDK's TeamDelete tool doesn't emit any special event. Detection
happens in `claude-session.ts` by scanning tool_use blocks:

```js
if (block.toolName === 'TeamDelete') {
    this._teamName = null
    this._detectedTeammates = []
    this.pendingTeammates.clear()
    this.teammateIdToToolUse.clear()
    this._teammateStatuses.clear()
    unwatchAllSubagents()
    this.send('session:team-deleted', {})
}
```

The renderer listens for `session:team-deleted` and calls `clearTeam()`
on the Zustand store, which sets `teamName: null`, clears `teammates`,
and resets `focusedAgentId`. The `AgentTabBar` component gates on
`teamName` being non-null, so it disappears automatically.

### Teammate ID routing

Messages from Patch B arrive with `teammate_id: "name@team"`. The
consumer maps this to `toolUseId` via `teammateIdToToolUse`:

```js
// Built when teammate is detected (Task tool result arrives):
const teammateId = `${pending.name}@${pending.teamName}`
this.teammateIdToToolUse.set(teammateId, toolUseId)

// Used when teammate message arrives:
resolveTeammateToolUseId(msg) {
    const teammateId = msg.teammate_id
    return this.teammateIdToToolUse.get(teammateId)
}
```

## What's NOT Changed

**Collection arrays** — `x.push(_6)` and `N.push(_6)` still operate on
assistant/user messages. Only `stream_event` skips them (via `continue`).

**im() state updates** — Still run for assistant/user messages before we
forward them. AppState is always up-to-date.

**R06 (stats tracking)** — Still called for all non-stream_event messages.

**Subagent-streaming patches** — Those handle the Task tool's sync/async
paths (progress callback + ZhA handler). This patch handles the teammate
path (KfY), which uses a different loop and has no progress callback.

**sm4 (idle notification)** — In the failure path, `sm4()` still fires
after our notification. It sends the idle notification to the team-lead's
inbox, which is unrelated to our stdout notification.

## Discovery Method

1. **JSONL fragmentation**: Observed 14 JSONL files in `subagents/` for
   a 2-agent 3-round debate. Traced `OR()` to find the `Qy()` fallback
   when `agentId` is not in the override.

2. **No event streaming**: Compared KfY's loop with the sync Task tool
   loop (patched by subagent-streaming). The sync loop has a progress
   callback `D`/`j` that pipes events to the parent. KfY has no such
   callback — it only calls `im()` to update AppState.

3. **B2 semicolon bug**: After applying patches, `bun run dev` showed
   "Query closed before response received" on `session:get-models`.
   `node --check cli.js` revealed `SyntaxError: Unexpected token 'if'`
   at the B2 injection. Root cause: missing `;` between `im(...)` and
   the injected `if(...)`.

4. **No completion notification**: After fixing B2, teammates streamed
   correctly but showed "running" forever in the UI. Added debug logs
   to `detectTaskMapping` and the `task_notification` handler. Confirmed
   `detectTaskMapping` correctly stored `taskIdMap` keys, but no
   `task_notification` system message ever arrived. Traced KfY's
   completion path — only `im()` calls, no stdout writes.

5. **C1/C2 semicolon bug**: First attempt at Patch C used `;` to
   terminate `process.stdout.write(...)`. `node --check` revealed
   `SyntaxError: Unexpected token ':'` — the `;` broke the comma
   expression (`return A, B, C`), making `{success:!0, messages:N}`
   dead code where `messages:N` is an invalid label.

6. **Verified full round-trip**: With Patches A+B+C applied, debug
   logs confirmed: `detectTaskMapping` stores agentId → toolUseId,
   `task_notification` arrives with matching task_id, lookup succeeds,
   UI updates to "completed".
